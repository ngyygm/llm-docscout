可以把整个研究设计成一个**“先验证环境，再验证策略，再验证 RL 是否真的带来增益”**的迭代过程，而不是一开始就搭一个复杂环境，然后直接训练。

真正需要建立的不是一套固定方案，而是一套**环境诊断与迭代准则**。

---

# 一、研究的核心问题要拆成三层

不要把所有问题都归结为“RL 效果好不好”。至少要分清三层：

## 第一层：任务本身是否可解

也就是：

> 文档中是否真的存在足够的信息，让 agent 通过 search 和 read 找到答案？

如果文档切分不合理、检索根本召回不到证据、问题需要跨文档推理但环境不支持，那么 RL 再强也没用。

这一层关注：

* 正确证据是否存在；
* 搜索工具是否能召回；
* section 是否可定位；
* observation 是否提供足够信息；
* 最短可行路径有多长。

## 第二层：策略是否可学

也就是：

> 给定一个可解环境，模型能否学会“搜什么、读哪个、什么时候停”？

这一层关注：

* action space 是否太复杂；
* observation 是否太长或太模糊；
* 小模型是否理解工具返回格式；
* 是否需要先 SFT；
* 是否存在明显的策略模式。

## 第三层：RL 是否真的有价值

也就是：

> 相比规则、提示词、SFT，RL 是否让 agent 更准确、更省读取、更会停止？

这一层才是真正评估强化学习方法。

---

# 二、正确的实验顺序

整个项目最好分为六个阶段。

---

# 阶段 0：先定义“什么叫做得好”

训练之前先锁定评价指标，否则后面奖励函数会不断跟着结果改，容易变成“为了证明有效而调指标”。

建议至少记录四类指标。

## 1. 结果正确性

* Answer Accuracy
* Exact Match
* F1
* LLM judge score
* 是否拒答正确

## 2. 证据质量

* Evidence Recall：有没有读到正确证据
* Evidence Precision：读到的内容中有多少真正相关
* Citation Correctness：引用内容是否支持答案
* Gold section hit rate

## 3. 检索效率

* search 次数
* read 次数
* 总读取 token 数
* 无关 section 数量
* 首次找到正确证据所需步骤
* 最终 episode 长度

## 4. 策略行为

* 是否过早停止
* 是否永不停止
* 是否重复搜索
* 是否重复读取
* 是否频繁 expand
* 搜索词是否逐步改善
* 是否发生 reward hacking

最后可以形成一个综合目标：

[
\text{Utility}
==============

\text{Answer Quality}
+
\lambda_1 \text{Evidence Quality}
---------------------------------

## \lambda_2 \text{Search Cost}

\lambda_3 \text{Read Cost}
]

但研究报告中不能只报一个总分，还必须拆开报。

---

# 阶段 1：先做“环境可解性实验”

这一阶段不要训练 RL。

目标是回答：

> 当前文档结构、检索工具和动作设计，理论上是否足够解决任务？

## 实验 1：Oracle Evidence 实验

直接把正确证据 section 给模型，只让它回答。

例如：

```text
question + gold evidence → answer
```

可能结果：

### 结果 A：准确率很高

说明只要找到正确证据，回答不是问题。

后续重点应放在检索策略。

### 结果 B：准确率仍然很低

说明不是检索问题，而可能是：

* 问题歧义；
* 答案标注错误；
* 证据不完整；
* 需要跨段推理；
* 模型能力不够；
* 答案评价标准有问题。

这时不应该训练 RL，而应先修数据和任务定义。

---

## 实验 2：Oracle Search Path 实验

给 agent 一个理想动作路径，例如：

```text
search("权限 同步")
read(candidate_1)
answer(...)
```

看模型能不能正确执行。

这可以验证：

* 工具格式是否容易理解；
* observation 是否清晰；
* 模型是否会正确引用候选；
* answer 格式是否稳定。

如果连理想路径都执行不好，说明 action schema 或 prompt 设计有问题。

---

## 实验 3：Retriever Upper Bound

独立评估搜索系统，不让 agent 参与。

计算：

```text
Recall@1
Recall@5
Recall@10
MRR
```

例如：

> Gold evidence 是否出现在搜索前 5 名？

可能结果：

### Recall@5 很高

说明基础检索已经够用，agent 主要负责候选选择和停止。

### Recall@20 才高

说明 snippet 排序不够，需要：

* query rewriting；
* hybrid search；
* reranker；
* 更好的切段；
* 元数据过滤。

### Recall@50 都很低

说明不是策略问题，而是文档索引或数据构造失败。

---

## 阶段 1 的通过条件

只有满足下面条件，才值得进入训练：

* 给 gold evidence 时，模型能正确回答；
* gold evidence 大多数情况下可被检索召回；
* 理想轨迹可以稳定执行；
* section ID 和位置是稳定的；
* 问题与证据标注质量可接受。

---

# 阶段 2：先跑无训练基线

不要立刻做 SFT 或 RL。先看原始模型自然表现。

至少跑四种基线。

## 基线 1：一次性 RAG

```text
search top-k → 全部放入上下文 → answer
```

这是最重要的对照组。

因为必须证明：

> 多步 agent 比普通一次性检索更有效，或者至少更省 token。

## 基线 2：固定流程 Agent

人为规定：

```text
search 一次
read 前 3 个
answer
```

这可以判断 agent 的主动决策是否真的必要。

## 基线 3：Prompted Agent

不训练，只用提示词让模型调用 search/read。

观察它是否已经有一定能力。

## 基线 4：Oracle

包括：

* oracle evidence；
* oracle path；
* oracle stopping。

它们代表不同层面的性能上限。

---

# 三、第一轮实验不要做大

第一轮应该是小规模、可诊断、容易人工检查的 pilot。

建议：

```text
500～2000 个问题
100～1000 篇文档
每题有 gold answer
每题有 gold evidence section
每题最好有难度标签
```

问题类型至少分开：

| 类型     | 例子         | 检验能力     |
| ------ | ---------- | -------- |
| 单段直接问题 | 同步多久       | 基础定位     |
| 同义改写   | 多久生效       | query 改写 |
| 干扰项问题  | 多篇文档都有“权限” | 候选选择     |
| 邻接信息问题 | 答案在下一段     | expand   |
| 跨段问题   | 条件和时限分开    | 多步阅读     |
| 跨文档问题  | 两份制度联合判断   | 多文档策略    |
| 无答案问题  | 文档中没有说明    | 正确停止或拒答  |

第一轮最好先只做：

* 单文档；
* 单证据段；
* 1～3 步可解决；
* 少量干扰项。

验证成功以后，再逐渐增加复杂度。

---

# 四、第一轮最重要的不是准确率，而是“失败分类”

每条失败轨迹都要归因。至少分成下面几类。

## 1. 检索失败

正确证据从未出现在候选中。

原因可能是：

* query 不好；
* 文档切分不合理；
* 索引不完整；
* 关键词不匹配；
* semantic search 失效。

## 2. 选择失败

正确证据已经在候选中，但 agent 没有读。

原因可能是：

* snippet 不够信息化；
* 候选太多；
* 排名混乱；
* agent 不理解相关性；
* read action 设计复杂。

## 3. 阅读失败

agent 读了正确 section，但没有识别答案。

原因可能是：

* section 太长；
  -答案埋得太深；
* 需要上下文；
* 模型阅读理解能力不足。

## 4. 停止失败

包括两种：

### 过早停止

还没有充分证据就回答。

### 过晚停止

已经找到答案却继续读很多内容。

## 5. 答案生成失败

证据已经充分，但回答错误或不规范。

## 6. 奖励投机

例如：

* 大量乱读以确保覆盖 gold evidence；
* 复制原文但不真正回答；
* 引用正确证据却生成错误答案；
* 为减少成本而直接猜；
* 利用答案长度、格式或标签漏洞骗分。

每轮实验之后都应该统计：

```text
所有失败中，检索失败占多少？
选择失败占多少？
停止失败占多少？
```

这比只看总准确率有价值。

---

# 五、数据环境应该怎样迭代

数据不是一次构建完成，而是要根据失败模式逐渐加难。

---

## 第一阶段：单证据、短路径

每个问题：

* 一个正确答案；
* 一个 gold section；
* 最短路径 1～2 个 read；
* 搜索结果中有明显干扰项。

目标是验证基本策略。

---

## 第二阶段：同义表达和检索困难

加入：

* 问题词和原文词不一致；
* 缩写与全称；
* 上位词与下位词；
* 隐含表述；
* 多个看起来相似的 section。

目标是训练 query formulation 和候选选择。

---

## 第三阶段：动态阅读

加入：

* 答案在相邻段落；
* 当前 section 只有部分线索；
* 需要 expand left/right；
* 标题与正文分离。

目标是训练“读多少”。

---

## 第四阶段：多证据推理

加入：

* 同一文档多个 section；
* 多篇文档联合；
* 条件例外；
* 时间版本冲突；
* 主规则与补充规则。

目标是训练证据整合。

---

## 第五阶段：无答案和不确定性

加入：

* 文档根本没有答案；
* 证据不足；
* 多份文档冲突；
* 答案只在过期文档中；
* 用户问题本身有歧义。

目标是防止 agent 总是强行回答。

---

## 一个重要原则：难度必须可测

不能只是随意增加“更长文档”。

每道题最好附带结构化标签：

```json
{
  "num_gold_sections": 2,
  "min_search_steps": 1,
  "min_read_steps": 2,
  "requires_expand": true,
  "cross_document": false,
  "has_distractors": true,
  "answerable": true
}
```

这样以后才能分析：

> RL 是在哪类任务上带来提升，而不是只看到平均分。

---

# 六、Reward 应该怎样迭代

不要一开始设计一个很复杂的 reward。

正确方法是逐层加。

---

## Reward V0：只奖励最终答案

```text
+1 answer correct
0 answer wrong
```

用途不是最终训练，而是观察模型自然策略。

通常可能出现：

* 模型直接猜；
* 搜索过程不稳定；
* reward 太稀疏；
* 学不到检索行为。

---

## Reward V1：加入证据奖励

```text
+10 answer correct
+3 evidence supports answer
-5 answer wrong
```

观察是否更愿意寻找正确证据。

风险：

> 模型可能大量阅读，确保碰到 gold evidence。

---

## Reward V2：加入动作成本

```text
-0.05 per search
-0.10 per read
-0.001 per token
```

观察读取量是否下降。

风险：

> 成本太高，模型会过早停止或直接猜答案。

---

## Reward V3：加入过程奖励

例如：

* 第一次召回 gold evidence：正奖励；
* 选择正确候选：正奖励；
* 重复读取：负奖励；
* 无效搜索词：轻微负奖励；
* 正确拒答：正奖励。

但过程奖励要谨慎，因为它容易把策略限制死。

如果你奖励：

```text
第一步必须 search
第二步必须 read
```

模型学到的可能只是固定模板，而不是真正策略。

---

## Reward V4：使用相对效率

与其固定扣 token，不如与可行路径相比：

[
R_{\text{efficiency}}
=====================

\frac{\text{oracle cost}}{\text{agent cost}}
]

或者：

[
R
=

## \text{quality}

\lambda
\max(0,\text{agent cost}-\text{reference cost})
]

这样不同难度问题的成本惩罚更公平。

否则简单问题和复杂问题用同一个 token budget 会不合理。

---

# 七、奖励函数迭代的准则

每次改 reward，都要问四个问题。

## 1. 它是否和最终目标一致？

例如“读到 gold section”不等于“证据真的支持答案”。

## 2. 模型有没有更容易的作弊方式？

例如：

* 读遍所有 section；
* 复制大量原文；
* 直接输出训练集高频答案；
* 不调用工具降低成本。

## 3. 奖励是否可被可靠计算？

如果 evidence score 完全依赖一个不稳定的 LLM judge，训练噪声可能很大。

## 4. 奖励是否过度约束路径？

同一个问题可能有多条合理搜索路径。

不要只奖励唯一的“标准动作序列”。

---

# 八、每一轮实验应该怎么跑

可以固定成下面的循环。

---

## Step 1：提出明确假设

例如：

> 加入 token cost 后，平均读取量会下降，但准确率下降不超过 2%。

或者：

> 动态 expand 能提高邻接证据问题的成功率。

不能只说：

> 看看效果怎么样。

---

## Step 2：只改变一个主要因素

例如只改：

* chunk size；
* search top-k；
* reward cost；
* observation history；
* 是否加入 expand；
* 是否加入 evidence reward。

不要同时改数据、模型、奖励和检索器，否则无法归因。

---

## Step 3：跑小规模实验

先跑：

```text
500～2000 条
2～3 个随机种子
```

观察趋势，而不是直接大规模训练。

---

## Step 4：检查自动指标

看：

* accuracy；
* evidence recall；
* average tokens；
* average steps；
* stop rate；
* invalid action rate。

---

## Step 5：人工看轨迹

每轮至少抽查：

```text
50～100 条完整轨迹
```

特别关注：

* 高奖励但行为明显错误；
* 答案正确但过程荒谬；
* 答案错误但检索正确；
* 成本异常低或异常高；
* 重复动作；
* reward hacking。

---

## Step 6：做失败归因

得到类似结果：

```text
检索失败：35%
候选选择失败：22%
停止过早：18%
答案生成失败：15%
其他：10%
```

---

## Step 7：只修主要瓶颈

例如：

### 如果检索失败最多

优先改：

* chunk；
* BM25 + embedding；
* query rewriting；
* reranking；
* 文档元数据。

不是改 RL。

### 如果选择失败最多

优先改：

* snippet；
* candidate 数量；
* action schema；
* SFT 数据；
* observation 格式。

### 如果停止失败最多

优先改：

* cost reward；
* confidence；
* answer action；
* no-answer 数据；
* stopping examples。

### 如果答案生成失败最多

优先改：

* answer format；
* evidence-conditioned generation；
* 独立 answer head；
* 模型能力；
* oracle evidence 训练。

---

# 九、一个推荐的实验矩阵

第一篇系统性研究可以围绕四条轴。

| 轴    | 取值                                              |
| ---- | ----------------------------------------------- |
| 数据难度 | 单证据 / 邻接证据 / 多证据 / 无答案                          |
| 环境设计 | 固定 chunk / 可 expand / 分层文档                      |
| 策略训练 | Prompt / SFT / SFT+RL                           |
| 奖励设计 | answer only / +evidence / +cost / adaptive cost |

核心实验不要一口气做全组合，可以逐层做：

```text
实验 A：环境可解性
实验 B：Prompt vs SFT
实验 C：SFT vs RL
实验 D：奖励消融
实验 E：数据难度泛化
实验 F：检索器或 chunk 鲁棒性
```

---

# 十、可能出现的几类结果

你最好在实验开始前就预想不同结果意味着什么。

---

## 结果 1：RL 同时提升准确率和效率

这是理想结果。

说明 RL 学到了更好的检索和停止策略。

还要进一步验证：

* 是否泛化到新文档；
* 是否泛化到更长文档；
* 是否只记住搜索词；
* 是否对检索器变化鲁棒。

---

## 结果 2：RL 降低成本，但准确率下降

说明成本惩罚过强，模型学会“少看少错”或直接猜。

可能改法：

* 降低 cost 权重；
* 只有答对时才计算效率奖励；
* 使用 accuracy constraint；
* 使用分阶段训练；
* 对简单题和复杂题设置不同 budget。

---

## 结果 3：RL 提高准确率，但读取量暴涨

说明证据奖励推动模型穷举。

可能改法：

* 对无关读取加惩罚；
* 对重复 read 加重惩罚；
* 使用 evidence precision；
* 限制最大候选；
* 引入相对成本。

---

## 结果 4：SFT 已经很好，RL 几乎无提升

这也是重要结果。

可能说明：

* 任务太简单；
* 示范路径已经接近最优；
* reward 信息不足；
* RL 只在分布外或复杂任务上有价值。

下一步应增加：

* 搜索歧义；
* 文档长度；
* 多跳任务；
* 无答案任务；
* 动态 budget；
* 检索器噪声。

---

## 结果 5：Prompt、SFT、RL 都不好

这通常不是模型问题，而是环境不可学。

重点检查：

* 检索 recall；
* observation 是否缺信息；
* tool schema 是否复杂；
* 数据是否存在标注错误；
* 最短路径是否太长；
* reward 是否稀疏。

---

## 结果 6：训练集很好，测试集崩溃

可能学到的是：

* 文档位置规律；
* 高频 query；
* 数据模板；
* 答案分布；
* 固定动作顺序。

需要按以下方式切分数据：

```text
按文档切分
按领域切分
按问题模板切分
按时间切分
```

不能只是随机切分 QA 对。

---

# 十一、训练环境设计的十条准则

可以把整个研究总结成十条硬原则。

## 1. 先验证可解性，再训练策略

检索不到证据时，不要怪 RL。

## 2. 先做 oracle，再做 baseline，再做 RL

必须知道问题出在哪一层。

## 3. 先简单后复杂

先单文档、单证据、短路径，再扩展。

## 4. 每题必须尽量有 gold evidence

否则无法区分“答对”和“查对”。

## 5. 评估答案、证据和成本三个维度

不能只看准确率。

## 6. Reward 要逐层增加

每加一项，都要检查行为变化和作弊方式。

## 7. 不奖励唯一标准路径

允许多条合理轨迹。

## 8. 每轮必须人工检查轨迹

高 reward 不代表策略正确。

## 9. 一次只改一个主要变量

否则实验不可解释。

## 10. 根据失败类型修改环境

不是所有失败都应该通过 RL 解决。

# 十二、最推荐的实际开工路线

可以直接按这个顺序开展。

## 第一轮：验证环境

* 构建 500～1000 道题；
* 每题标注答案和证据；
* 做 oracle evidence；
* 做 retriever recall；
* 检查 action 能否稳定执行。

## 第二轮：跑基线

* 一次性 RAG；
* 固定 search-read-answer；
* prompt agent；
* oracle path。

## 第三轮：构建示范轨迹

* 用规则生成简单轨迹；
* 用强模型生成复杂轨迹；
* 人工抽查；
* 做 SFT。

## 第四轮：做最简单 RL

Reward 只包含：

```text
answer correctness
+ evidence correctness
- small action cost
```

不要一开始加入十几个 reward 项。

## 第五轮：做行为诊断

重点看：

* 过早停止；
* 过度读取；
* 重复搜索；
* 搜索词退化；
* reward hacking。

## 第六轮：根据主要失败迭代

* 检索失败 → 改索引与数据；
* 选择失败 → 改 observation 与 SFT；
* 停止失败 → 改 cost 与负样本；
* 推理失败 → 改证据组织与模型。

## 第七轮：增加难度

逐步加入：

* 同义搜索；
* 干扰文档；
* 动态 expand；
* 多跳证据；
* 无答案；
* 文档版本冲突。

# 最终可以形成一条研究主线

这项研究最有价值的地方，不只是提出某个 reward，而是研究：

> **怎样构建一个可学习、可诊断、可逐渐加难的信息获取环境，以及不同环境与奖励设计如何影响 agent 的检索、阅读和停止行为。**

整套实验逻辑可以浓缩成：

```text
先证明答案可从文档中得到
→ 再证明工具能把证据找到
→ 再证明模型能学会基本轨迹
→ 再用 RL 优化策略
→ 观察行为失败
→ 根据失败类型修改数据、工具或奖励
→ 逐渐提高任务难度
```

最重要的准则是：

> **任何一次性能下降，都先判断是数据问题、检索问题、环境问题、策略问题，还是奖励问题；不要默认都用更复杂的 RL 解决。**
